CH 01 참고

프롬프트 구조화 하기

좋은 프롬프트는 문장 몇 개가 아니라 구조다. Markdown은 읽기 쉬운 섹션을 만들고, XML은 역할·맥락·예시의 경계를 엄격하게 나눠서 모델이 혼동하지 않게 한다.
왜 구조가 성능을 바꾸나

AWS, Google, Anthropic 공식 가이드는 공통적으로 명확한 섹션 구분과 일관된 포맷을 권한다. 구조가 잡히면 모델이 역할, 맥락, 작업, 제약을 덜 혼동하고 긴 입력에서도 집중점을 유지하기 쉬워진다.

Markdown

읽기 쉽고 빠르게 설계할 때

헤더, 불릿, 코드 블록으로 역할·요청·출력 형식을 나누는 방식이다. Gemini 같은 대화형 실습에서 빠르게 쓰고 고치기에 가장 가볍다.

  • # / ##로 섹션 계층을 만든다.
  • -로 요구사항과 체크리스트를 분리한다.
  • ```로 예시 데이터나 출력 샘플을 본문과 분리한다.
XML

경계와 위계를 엄격하게 나눌 때

instructions, context, examples, input 같은 태그로 블록을 분리하는 방식이다. 긴 문서, 복잡한 지시, 여러 예시가 섞인 프롬프트에서 특히 강하다.

  • <instructions>, <context>, <example>처럼 역할을 태그로 분리한다.
  • 중첩 구조로 문서와 메타데이터의 위계를 유지할 수 있다.
  • 특히 Claude 계열 문서에서 권장하는 방식이다.

Markdown 패턴

왼쪽에서 Markdown을 수정하면 오른쪽 미리보기가 함께 바뀐다. 역할, 요청, 출력 형식, 제약을 분리하는 가장 가벼운 구조화 방식이다.

Markdown 입력
렌더링 결과

역할

당신은 LG전자 가전 사업부의 전략 컨설턴트입니다.


요청

아래 보고서를 분석하여 핵심 이슈 3가지를 추출해주세요.


출력 형식

  • 이슈명: 한 줄 요약
  • 현황: 데이터 기반 설명
  • 제안: 실행 방안

제약 조건

  • 경영진 보고용 격식체
  • 확인되지 않은 사실은 추정으로 쓰지 않기
XML 패턴
Example

태그로 역할과 맥락을 분리

<system_instruction>
당신은 고객 전략팀을 돕는 분석가입니다.
제공된 자료 안에서만 근거를 찾아 답변하세요.
</system_instruction>

<context>
[시장 보고서, 고객 인터뷰, 경쟁사 자료]
</context>

<user_query>
2026년 핵심 리스크 3가지를 정리해줘.
</user_query>
Why It Works

복잡한 입력에서 강한 이유

  • 지시문과 데이터의 경계가 명확해져서 모호성이 줄어든다.
  • 예시와 실제 입력을 태그로 구분해 few-shot 품질을 올리기 쉽다.
  • 긴 문서, 다단계 분석, 에이전트용 system prompt처럼 위계가 중요한 작업에 잘 맞는다.

Anthropic은 XML 태그가 instructions, context, examples, input을 섞지 않게 도와준다고 설명하고, Google도 XML-style tags나 Markdown headings처럼 일관된 구조를 권장한다.

최근 흐름
실무는 단순한 Prompt Engineering에서 끝나지 않고, 좋은 자료 묶음을 설계하는 Context Engineering, 그리고 그 구조를 반복 가능한 작업 방식으로 만드는 Workflow Design으로 확장되고 있다. 즉, 좋은 한 문장보다 좋은 구조와 좋은 맥락이 더 중요해지는 방향이다.
참고한 공식 가이드
  • Anthropic Prompting Best Practices — XML 태그로 instructions, context, examples를 분리하는 방식을 권장합니다.
  • Google Gemini Prompting Strategies — XML-style tags 또는 Markdown headings처럼 일관된 구조를 사용하라고 안내합니다.
  • AWS Nova Prompt Focus — Markdown, XML 같은 구조로 섹션을 나누면 모델이 특정 블록에 더 잘 집중할 수 있다고 설명합니다.
