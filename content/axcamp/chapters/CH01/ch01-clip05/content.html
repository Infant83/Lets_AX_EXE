CH 03 참고

Gems 소개: 프롬프트 구조화하기

좋은 프롬프트는 문장 몇 개보다 구조가 더 중요합니다. 그리고 잘 만든 구조는 Gems에 저장해 회의록 요약기, 심층 리서치 프롬프트 작성기 같은 메타프롬프트 생성기로 재사용할 수 있습니다.
왜 구조가 성능을 바꾸나

AWS, Google, Anthropic 공식 가이드는 공통적으로 명확한 섹션 구분과 일관된 포맷을 권합니다. 구조가 잡히면 모델이 역할, 맥락, 작업, 제약을 덜 혼동하고 긴 입력에서도 집중점을 유지하기 쉬워집니다.

Markdown

읽기 쉽고 빠르게 설계할 때

헤더, 불릿, 코드 블록으로 역할·요청·출력 형식을 나누는 방식입니다. Gemini 같은 대화형 실습에서 빠르게 쓰고 고치기에 가장 가볍습니다.

  • # / ##로 섹션 계층을 만듭니다.
  • -로 요구사항과 체크리스트를 분리합니다.
  • ```로 예시 데이터나 출력 샘플을 본문과 분리합니다.
XML

경계와 위계를 엄격하게 나눌 때

instructions, context, examples, input 같은 태그로 블록을 분리하는 방식입니다. 긴 문서, 복잡한 지시, 여러 예시가 섞인 프롬프트에서 특히 강합니다.

  • <instructions>, <context>, <example>처럼 역할을 태그로 분리합니다.
  • 중첩 구조로 문서와 메타데이터의 위계를 유지할 수 있습니다.
  • 특히 Claude 계열 문서에서 권장하는 방식입니다.
언제 무엇을 쓰나
구분MarkdownXML
강점읽기 쉽고 작성이 빠릅니다구조와 경계가 엄격해 복잡한 입력에 강합니다
잘 맞는 상황일반적인 대화형 프롬프트, 요약, 초안 작성, 간단한 보고서 지시긴 문서 분석, 복수 예시, 멀티파트 지시, 도구/컨텍스트 분리
오늘 수업 연결Gemini, NotebookLM, AI Studio Build 초반 실습Claude 스타일 구조화, 긴 컨텍스트, 에이전트용 instruction 설계
핵심 포인트사람과 모델이 모두 읽기 쉬운 포맷모델이 어떤 블록이 데이터이고 어떤 블록이 지시인지 덜 헷갈림
Markdown 패턴

왼쪽에서 Markdown을 수정하면 오른쪽 미리보기가 함께 바뀝니다. 역할, 요청, 출력 형식, 제약을 분리하는 가장 가벼운 구조화 방식입니다.

Markdown 입력
렌더링 결과

역할

당신은 LG전자 가전 사업부의 전략 컨설턴트입니다.


요청

아래 보고서를 분석하여 핵심 이슈 3가지를 추출해주세요.


출력 형식

  • 이슈명: 한 줄 요약
  • 현황: 데이터 기반 설명
  • 제안: 실행 방안

제약 조건

  • 경영진 보고용 격식체
  • 확인되지 않은 사실은 추정으로 쓰지 않기
XML 패턴
Example

태그로 역할과 맥락을 분리

<system_instruction>
당신은 고객 전략팀을 돕는 분석가입니다.
제공된 자료 안에서만 근거를 찾아 답변하세요.
</system_instruction>

<context>
[시장 보고서, 고객 인터뷰, 경쟁사 자료]
</context>

<user_query>
2026년 핵심 리스크 3가지를 정리해줘.
</user_query>
Why It Works

복잡한 입력에서 강한 이유

  • 지시문과 데이터의 경계가 명확해져서 모호성이 줄어듭니다.
  • 예시와 실제 입력을 태그로 구분해 few-shot 품질을 올리기 쉽습니다.
  • 긴 문서, 다단계 분석, 에이전트용 system prompt처럼 위계가 중요한 작업에 잘 맞습니다.

Anthropic은 XML 태그가 instructions, context, examples, input을 섞지 않게 도와준다고 설명하고, Google도 XML-style tags나 Markdown headings처럼 일관된 구조를 권장합니다.

왜 Gems를 만드는가
구분일반 채팅Gems
시작 비용대화마다 역할과 배경을 다시 설명역할, 톤, 출력 형식을 한 번 저장
일관성질문이 바뀌면 답변 스타일도 흔들릴 수 있음같은 업무는 같은 기준으로 답하도록 고정 가능
확장성개인 단위 실험에 적합팀 단위 템플릿, 비서형 워크플로우로 확장 가능
오늘 실습 연결비즈니스 프롬프트를 직접 써 봄잘 된 프롬프트를 비서처럼 재사용
한 줄 요약
일반 채팅이 그때그때 지시하는 방식이라면, Gems는 직무기술서가 정의된 전담 비서를 배치하는 방식입니다.
만들기 3단계

세 장 모두 클릭하면 크게 볼 수 있고, 확대 모달에서 좌우로 넘기며 단계별로 확인할 수 있습니다.

메타프롬프트 Gem 예시

구조화된 프롬프트를 한 번 더 추상화하면, 프롬프트를 대신 설계해 주는 Gem을 만들 수 있습니다. 아래 두 예시는 회의록 요약과 심층 리서치처럼 반복되는 업무에서 바로 재사용할 수 있는 메타프롬프트 생성기입니다.

Gem 이름무엇을 만들어주나언제 유용한가
회의록 요약 프롬프트 생성기회의 주제, 참석자, 목적을 받아 임원 보고용 회의록 요약 프롬프트를 설계회의가 자주 반복되고, 참석자나 산출 형식만 조금씩 바뀔 때
심층 리서치 프롬프트 작성기조사 주제, 독자, 의사결정 목적을 받아 Deep Research용 구조화 프롬프트를 설계전략 보고서, 시장 조사, 경쟁사 비교 리서치를 빠르게 시작할 때
회의록 요약 프롬프트 생성 Gem 요청
당신은 기업 임원회의 지원용 메타프롬프트 설계자입니다. 내가 회의 주제, 참석자, 회의 목적, 원하는 산출물을 입력하면 Gemini에서 바로 사용할 수 있는 **회의록 요약 프롬프트**를 만들어 주세요. 반드시 아래 구조로 출력하세요. # 역할 # 사용자가 입력할 변수 # 최종 프롬프트 # 출력 형식 # 후속 질문 3개 프롬프트 설계 조건: - 경영진 보고용 격식체 - 참석자, 핵심 발언, 의사결정, 미해결 이슈, 액션 아이템 포함 - 표와 불릿을 함께 활용 - 확인되지 않은 내용은 추정하지 않기 예시 입력: - 회의 주제: AX 추진 점검회의 - 참석자: CEO, CIO, 사업부장 - 목적: 핵심 이슈 요약 및 다음 액션 정리
심층 리서치 프롬프트 작성 Gem 요청
당신은 심층 리서치용 메타프롬프트 작성 Gem입니다. 내가 조사 주제, 독자, 의사결정 목적, 원하는 산출 형식을 입력하면 Gemini Deep Research나 다른 리서치 모델에 바로 넣을 수 있는 **고품질 조사 프롬프트**를 만들어 주세요. 반드시 아래 구조로 출력하세요. # 리서치 목적 # 조사 범위 # 포함할 하위 질문 # 요구 데이터와 출처 기준 # 최종 프롬프트 # 보고서 목차 초안 프롬프트 설계 조건: - 최신 데이터 우선 - 출처 표기 필수 - 경쟁사 비교 가능하도록 설계 - 마지막에 경영진 시사점 포함 예시 입력: - 주제: 글로벌 AI 가전 시장에서 LG의 포지셔닝 전략 - 독자: CEO 및 사업부 리더 - 목적: 투자 우선순위 결정 - 형식: 1페이지 요약 + 상세 부록
최근 흐름
실무는 단순한 Prompt Engineering에서 끝나지 않고, 좋은 자료 묶음을 설계하는 Context Engineering, 그리고 그 구조를 반복 가능한 작업 방식으로 만드는 Workflow Design으로 확장되고 있다. 즉, 좋은 한 문장보다 좋은 구조와 좋은 맥락이 더 중요해지는 방향입니다.
참고한 공식 가이드
  • Anthropic Prompting Best Practices — XML 태그로 instructions, context, examples를 분리하는 방식을 권장합니다.
  • Google Gemini Prompting Strategies — XML-style tags 또는 Markdown headings처럼 일관된 구조를 사용하라고 안내합니다.
  • AWS Nova Prompt Focus — Markdown, XML 같은 구조로 섹션을 나누면 모델이 특정 블록에 더 잘 집중할 수 있다고 설명합니다.
